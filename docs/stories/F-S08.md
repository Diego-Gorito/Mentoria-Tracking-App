## Story

Como **ops (Felix)**, eu quero **estender o compose `tracking-backup` pra incluir snapshot Redis das keys `gtm:*` no MinIO daily**, pra que **audit log + installations sobrevivam a desastre no volume Redis** (LGPD-relevant, retenção 30d alinhada pg backup).

## Acceptance Criteria

### AC-1 — Compose modificado adiciona service Redis snapshot

- **Given** que `tracking-backup-compose.yml` já faz pg_dump 03h BRT
- **When** Felix modifica YAML
- **Then** existe service `redis-snapshot` (ou step adicional no existing service `tracking-backup`) executando snapshot Redis no mesmo cron 03h.

### AC-2 — Cron 03h BRT alinhado pg backup

- **Given** que ADR-0008a §6.1 R7 requer "Retenção 30d alinhada com pg backup"
- **When** Felix configura cron
- **Then** cron expression `0 3 * * *` em timezone `America/Sao_Paulo` (env `TZ=America/Sao_Paulo` no container OR cron expression ajustada pra 06h UTC).

### AC-3 — Snapshot só keys gtm:*

- **Given** que `metabase:*` e `track:dedup:*` NÃO devem entrar no backup
- **When** script executa
- **Then** abordagem (Felix decide simplicidade vs robustez):
  - **Opção A (simpler)**: `redis-cli BGSAVE` full RDB + extrair keys gtm:* via script Python/Lua → upload filtered RDB
  - **Opção B (robust)**: `redis-cli --scan --pattern 'gtm:*' | xargs -L1 redis-cli DUMP > backup.txt` → parsing manual
  - **Opção C (clean)**: usar `redis-cli MIGRATE` pra Redis temp container só com keys gtm:* + BGSAVE nele
- Resultado final: arquivo `gtm-snapshot.rdb` (ou `.txt`) contém apenas as keys `gtm:*`.

### AC-4 — Upload MinIO com nomenclatura datada

- **Given** que MinIO bucket `tracking-backups` já existe (CLAUDE.md stack)
- **When** snapshot pronto
- **Then** upload pra path `tracking-backups/redis/gtm-snapshot-<YYYY-MM-DD>.rdb` usando `mc` (MinIO client) ou `aws s3 cp` (S3-compatible API):
  ```bash
  mc cp /tmp/gtm-snapshot.rdb minio/tracking-backups/redis/gtm-snapshot-$(date +%Y-%m-%d).rdb
  ```

### AC-5 — Retention 30d

- **Given** que MinIO bucket pg_dump retention é 30d
- **When** snapshot novo é uploaded
- **Then** script remove snapshots >30 dias:
  ```bash
  mc rm --recursive --older-than 30d minio/tracking-backups/redis/
  ```
  OU usar MinIO lifecycle policy (mais robusto, configura no bucket).

### AC-6 — Healthcheck sanity size

- **Given** que Redis vazio retorna RDB ~80 bytes (header only)
- **When** snapshot finaliza
- **Then** script valida `size > 100 bytes` (sanity). Se <100: log warning + continua (não bloqueia upload — pode ser estado válido durante setup).

### AC-7 — Doc runbook restore procedure

- **Given** que recovery é raro, precisa doc clara
- **When** Felix finaliza
- **Then** `docs/runbook-ops.md` (criar se não existir) tem seção:
  ```markdown
  ## Restore gtm:* snapshot

  1. Download do MinIO: `mc cp minio/tracking-backups/redis/gtm-snapshot-YYYY-MM-DD.rdb /tmp/`
  2. Spin up Redis temp container: `docker run -d --name redis-restore -v /tmp:/tmp redis:7-alpine`
  3. Restore: `docker exec redis-restore redis-cli --pipe < /tmp/gtm-snapshot.rdb`
     OU: `cat /tmp/gtm-snapshot.rdb | docker exec -i redis-restore redis-cli --pipe`
  4. Verify: `docker exec redis-restore redis-cli KEYS 'gtm:*'` → lista keys
  5. Re-import pro Redis prod (manual ou script):
     - Para cada key: `redis-cli -h <restore> DUMP <key> | redis-cli -h redis RESTORE <key> 0`
  ```

### AC-8 — Smoke manual

- **Given** que validation pré-deploy é mandatory
- **When** Felix roda manualmente após config
- **Then**:
  1. `docker compose -f tracking-backup-compose.yml run --rm redis-snapshot` executa 1 ciclo
  2. MinIO browser mostra `gtm-snapshot-2026-05-25.rdb` no path correto
  3. Size >100 bytes
  4. Felix faz dry-run restore em Redis temp container → keys `gtm:*` aparecem

## Edge Cases

1. **Redis offline durante cron:** script tenta redis-cli, falha. Log warning, sai sem upload. Aceita perda de 1 dia (próxima execução cobre).
2. **MinIO offline:** upload falha. Script retém RDB local em `/tmp/gtm-snapshot-failed-<date>.rdb` por 7 dias pra retry manual.
3. **Disk full em /tmp:** BGSAVE pode falhar. Felix monitora KV8 disk via Easypanel.
4. **Keys gtm:* em mid-write durante BGSAVE:** RDB snapshot é point-in-time consistent (Redis garantee). OK.
5. **Restore corrupts prod:** procedure AC-7 explicit usa **temp container**, NÃO restore direto prod. Diego pesa qual keys merge.
6. **Filter `gtm:*` via SCAN race:** se keys são criadas durante SCAN, podem ser missadas. Aceito MVP (1 dia perda no pior caso).
7. **MinIO lifecycle policy vs mc rm --older-than:** lifecycle é melhor (atomicity). Felix decide se vale complexity config vs simple `mc rm` no script.
