## Story

Como **dev backend (Dex)**, eu quero **uma camada de storage Redis port-adapter implementada por trás da interface `IGtmStorage`**, pra que **endpoints, validator e audit helper possam ler/escrever installations sem acoplar lógica de business com Redis específico** — e Onda 1.5 troque por Supabase sem refactor de callers.

## Acceptance Criteria (Given/When/Then)

### AC-1 — Interface `IGtmStorage` definida completa

- **Given** que ADR-0008a §2.5 define 12 métodos (CRUD accounts + CRUD installations + audit + lock)
- **When** Dex cria `tracking-api/src/lib/storage/IStorage.ts`
- **Then** o arquivo exporta uma interface `IGtmStorage` contendo TODOS os 12 métodos com types corretos (assinaturas idênticas ao ADR), e o TypeScript compila sem `tsc --noEmit` errors quando esse arquivo é importado por um stub.

### AC-2 — Branded types previnem mistura acidental de IDs

- **Given** que IDs de Account vs Installation são strings UUID v4 mas com semântica distinta
- **When** Dex define em `tracking-api/src/lib/storage/types.ts` os branded types `AccountId`, `InstallationId`, `TenantId`, `ISO8601` per ADR-0008a §2.4 (linhas 86-89)
- **Then** uma tentativa de passar um `InstallationId` num parâmetro que espera `AccountId` **falha em compile-time** (`tsc --noEmit` retorna error TS2345 incompatible types).

### AC-3 — `RedisGtmStorage` usa key conventions canônicas

- **Given** que ADR-0008a §2.3 define 9 padrões de keys (`gtm:account:*`, `gtm:install:*`, `gtm:audit:*`, `gtm:lock:install:*`)
- **When** Dex implementa `tracking-api/src/lib/storage/RedisGtmStorage.ts` usando `ioredis`
- **Then** cada método mapeia pra exatamente as keys listadas no ADR: `createAccount` usa `gtm:account:<id>` HASH + adiciona ao `gtm:account:list` SET, `createInstallation` usa `gtm:install:<id>` HASH + `gtm:install:by_site:<sha1(domain)>` STRING, `appendAudit` usa `gtm:audit:<installation_id>` LIST via `LPUSH` + `LTRIM 0 999`, e nenhuma key fora do prefix `gtm:*` é tocada (evita colisão com `metabase:*` e `track:dedup:*`).

### AC-4 — Idempotência via sha1(site_domain)

- **Given** que reinstall do mesmo domain NÃO deve duplicar installation_id
- **When** chamadas concorrentes ou sequenciais `createInstallation({ site_domain: "zerohum.com.br", ... })` ocorrem N vezes
- **Then** todas retornam o **mesmo `installation_id`**, validado por: 1ª chamada cria `gtm:install:by_site:<sha1("zerohum.com.br")>` apontando pra novo UUID; 2ª+ chamadas leem essa key, retornam o existing `GtmInstallation` (não criam novo HASH).

### AC-5 — Distributed lock via SET NX EX

- **Given** que multi-replica `tracking-api` pode disparar 2 deploys concurrent na mesma installation
- **When** `acquireLock(installation_id, 60)` é chamado 2× em paralelo (ms-window)
- **Then** apenas 1 chamada retorna `true`; a 2ª retorna `false` até `releaseLock(installation_id)` executar OU TTL 60s expirar. Implementação usa `SET gtm:lock:install:<id> <random-value> NX EX 60` (Redis atomic) — re-tentativa em <50ms confirma comportamento.

### AC-6 — Factory `getStorage()` com feature flag

- **Given** que Onda 1.5 vai trocar Redis por Supabase sem tocar callers
- **When** Dex cria `tracking-api/src/lib/storage/index.ts` exportando `getStorage(backend?: 'redis' | 'supabase'): IGtmStorage`
- **Then** `getStorage(env.STORAGE_BACKEND ?? 'redis')` retorna instância de `RedisGtmStorage` no MVP F; chamada com `'supabase'` lança erro `"Supabase storage backend not implemented in MVP F — see ADR-0008a §4"`.

### AC-7 — Unit tests cobrem CRUD + idempotency + lock + persistence

- **Given** que `redis-memory-server` permite testar Redis sem container externo
- **When** Dex roda `vitest run` em `tracking-api/src/lib/storage/__tests__/RedisGtmStorage.test.ts`
- **Then** 7 grupos de testes passam:
  1. `createAccount` + `getAccount` + `listAccounts` round-trip
  2. `updateAccount` patch funcional, fields ausentes preservam valor original (não viram undefined)
  3. `createInstallation` 2× mesmo domain → mesmo ID (AC-4)
  4. `appendAudit` + `listAudit limit=10` retorna últimos 10 (LIFO via LPUSH)
  5. `acquireLock` true → 2ª acquireLock false → releaseLock → 3ª acquireLock true
  6. TTL lock auto-expira em <60s (test usa `ttl_sec=2` pra velocidade)
  7. Restart in-memory Redis simulado + AOF replay — keys `gtm:*` persistem (mocka via `fakeRedis.disconnect() + reconnect`)

## Tech Notes

**Decisões prévias herdadas:**
- ADR-0008a §2.2 escolheu **Redis** (não JSON file) — infra já existe em KV8 (`infra/easypanel/redis-compose.yml`).
- ADR-0008a §2.3 define key conventions canônicas — **NÃO inventar prefixes novos**.
- ADR-0008a §2.4 define interfaces TypeScript completas — copiar literal.
- Port pattern espelha ADR-0008 §3.3 (provider adapter já estabelecido no repo).

**Patterns a usar:**
- `ioredis` client (single connection, reuse via singleton em `src/lib/redis.ts` se já existir; criar se não).
- `crypto.createHash('sha1').update(domain).digest('hex')` pra idempotency key suffix.
- `crypto.randomUUID()` pra IDs (UUID v4 nativo Node 19+).
- `Object.freeze()` em config constants (CONFIG, PREFIXES) pra prevenir mutação acidental.
- Multi/EXEC só onde MUITO necessário — preferir atomic ops single (HSET, SADD).

**LoC estimado:** ~250 (interface 60 + impl 130 + types 30 + tests 30).

**Sem migrations DB** — MVP F throwaway code (Manifesto 22/05 vigente, ADR-0008a §1).

## Edge Cases

1. **Redis container restart mid-write:** `appendAudit` LPUSH pode falhar parcialmente. Mitigação: catch error + retry 1× (NÃO loop infinito); se 2ª falha, log Docker stdout (Easypanel retém 7d per ADR-0008 §3.7). Não compromete idempotência (audit é append-only, perda de 1 entry é aceitável MVP).
2. **Redis OOM (`maxmemory` policy `allkeys-lru`):** keys `gtm:*` podem ser evicted se Redis estourar 256MB. Mitigação atual: ADR-0008a §6.2 R9 estima <2MB MVP, folga grande. Story F-S08 cobre backup MinIO daily.
3. **SHA1 collision em site_domain:** SHA1 tem 160-bit space, collision astronomicamente improvável pra 10-100 domains. Não documentar como risk.
4. **Lock acquire concurrent sem releaseLock (worker crash):** TTL 60s auto-libera. Pattern Redlock NÃO necessário MVP (single Redis instance, Diego único user).
5. **Race condition na 1ª createInstallation com idempotency:** 2 calls concurrent com mesmo domain podem ambas chegar antes do `gtm:install:by_site:` set. Mitigação: usar `SET NX` pra criar o by_site key — perdedor lê o vencedor e retorna o ID dele.
