## Story

Como **dev backend (Dex)**, eu quero **helpers `sealEncrypt`/`sealDecrypt` baseados em libsodium sealed box**, pra que **tokens Hostinger + WP admin passwords sejam criptografados antes de gravar no Redis** — sem Vault disponível no MVP, sem expor plaintext no `redis-cli`.

## Acceptance Criteria (Given/When/Then)

### AC-1 — Módulo `crypto.ts` expõe sealEncrypt + sealDecrypt

- **Given** que ADR-0008a §3.2 define 2 funções públicas com signatures fixas
- **When** Dex cria `tracking-api/src/lib/storage/crypto.ts`
- **Then** o módulo exporta:
  ```ts
  export async function sealEncrypt(plaintext: string, publicKeyBase64: string): Promise<string>
  export async function sealDecrypt(cipherBase64: string, publicKeyBase64: string, secretKeyBase64: string): Promise<string>
  ```
  ambas com types corretos e doc-comment `@see ADR-0008a §3.2`.

### AC-2 — Implementação usa `libsodium-wrappers` (não wrappers caseiros)

- **Given** que NaCl/libsodium é battle-tested e o npm package `libsodium-wrappers` é mantido
- **When** Dex implementa sealEncrypt + sealDecrypt
- **Then** o código usa **exclusivamente** `sodium.crypto_box_seal` e `sodium.crypto_box_seal_open` (NÃO implementa AES-GCM manual, NÃO usa `crypto` nativo Node — ver ADR-0008a §3.1 tabela: opções C e D rejeitadas). Dep `libsodium-wrappers@^0.7` adicionada ao `tracking-api/package.json`.

### AC-3 — Env vars novos validados em runtime

- **Given** que sem keys, sealEncrypt não tem como cifrar
- **When** `tracking-api` inicializa
- **Then** existe um `assertEnv()` (em `src/lib/env.ts` ou `crypto.ts` mesmo) que verifica presence de `STORAGE_ENCRYPTION_PUBLIC_KEY` e `STORAGE_ENCRYPTION_SECRET_KEY` no boot — se algum ausente, processo falha com error claro `"STORAGE_ENCRYPTION_PUBLIC_KEY ausente — ver ADR-0008a §3.2 pra geração de keypair"` (NÃO inicia silenciosamente com placeholder).

### AC-4 — Script gerador de keypair pronto pra Diego rodar

- **Given** que Diego precisa gerar UM par de chaves e colar no Easypanel
- **When** Dex cria `tracking-api/scripts/generate-libsodium-keypair.ts`
- **Then** o script:
  1. Importa `libsodium-wrappers` + aguarda `sodium.ready`
  2. Chama `sodium.crypto_box_keypair()` → obtém `{ publicKey, privateKey }`
  3. Imprime no stdout em formato copiável:
     ```
     STORAGE_ENCRYPTION_PUBLIC_KEY=<base64 publicKey>
     STORAGE_ENCRYPTION_SECRET_KEY=<base64 secretKey>
     ```
  4. Imprime aviso final: `"Cole essas 2 linhas no Easypanel → tracking-api → Env tab. NÃO commitar."`
  5. Roda com `pnpm tsx tracking-api/scripts/generate-libsodium-keypair.ts` (sem build step).

### AC-5 — Roundtrip test passa

- **Given** que cifrar+decifrar a mesma string deve produzir o original
- **When** rodar vitest com:
  ```ts
  const { publicKey, privateKey } = await generateKeypair();
  const cipher = await sealEncrypt('hostinger-token-test', publicKey);
  const plain = await sealDecrypt(cipher, publicKey, privateKey);
  ```
- **Then** `plain === 'hostinger-token-test'` (assertion exact equality, sem trim/lowercase tricks).

### AC-6 — Decrypt com chave errada rejeita (NaCl semantics)

- **Given** que sealed box garante autenticidade — sem secret key correta, abrir cipher falha
- **When** rodar:
  ```ts
  const kpA = await generateKeypair();
  const kpB = await generateKeypair();
  const cipher = await sealEncrypt('secret', kpA.publicKey);
  // tenta decrypt com keypair B (chave errada)
  await sealDecrypt(cipher, kpB.publicKey, kpB.privateKey);
  ```
- **Then** a chamada **rejeita** (`throw`) — NÃO retorna plaintext silenciosamente, NÃO retorna empty string, NÃO retorna garbage. Test valida via `expect(...).rejects.toThrow()`.

## Tech Notes

**Decisões prévias herdadas:**
- ADR-0008a §3.1 escolheu **libsodium sealed box (opção A)** — rejeitando age (B), AES-GCM nativo (C, error-prone) e plaintext (D, LGPD-fail).
- Sealed box = `crypto_box_seal` — anonymous sender, recipient-only decrypt. Perfeito pra storage at-rest.
- Env vars `STORAGE_ENCRYPTION_PUBLIC_KEY` + `STORAGE_ENCRYPTION_SECRET_KEY` ambos em base64 (32 bytes cada).

**Patterns a usar:**
- `await sodium.ready` antes da 1ª operação (boot init). Singleton flag `initialized = false` previne race re-init.
- `sodium.base64_variants.ORIGINAL` (não URL-safe) — consistente com geração no script.
- `sodium.to_string(plain)` converte Uint8Array de volta pra string UTF-8.
- NÃO usar `Buffer.from(base64)` mix com libsodium — usa `sodium.from_base64`.

**LoC estimado:** ~80 total (crypto.ts ~50 + script ~25 + tests ~80 — sim, tests > impl, é correto).

**Performance:**
- sealEncrypt 1 token (~40 chars): ~5-10ms.
- sealDecrypt 1 cipher: ~3-7ms.
- Boot init libsodium: ~15-30ms (only once).
- Aceitável: chamadas só em CRUD account (~10/dia MVP).

## Test Plan

### Unit (vitest)

```ts
// __tests__/crypto.test.ts (estrutura)

import { describe, it, expect, beforeAll } from 'vitest';
import sodium from 'libsodium-wrappers';
import { sealEncrypt, sealDecrypt } from '../crypto';

let kpA: { publicKey: string; privateKey: string };
let kpB: { publicKey: string; privateKey: string };

async function gen() {
  await sodium.ready;
  const kp = sodium.crypto_box_keypair();
  return {
    publicKey: sodium.to_base64(kp.publicKey, sodium.base64_variants.ORIGINAL),
    privateKey: sodium.to_base64(kp.privateKey, sodium.base64_variants.ORIGINAL),
  };
}

beforeAll(async () => {
  kpA = await gen();
  kpB = await gen();
});

describe('crypto sealed box', () => {
  it('roundtrip: sealEncrypt + sealDecrypt retorna original', async () => {
    const plain = 'hostinger-api-token-abc123xyz';
    const cipher = await sealEncrypt(plain, kpA.publicKey);
    expect(await sealDecrypt(cipher, kpA.publicKey, kpA.privateKey)).toBe(plain);
  });

  it('rejeita decrypt com keypair diferente', async () => {
    const cipher = await sealEncrypt('secret', kpA.publicKey);
    await expect(
      sealDecrypt(cipher, kpB.publicKey, kpB.privateKey)
    ).rejects.toThrow();
  });

  it('lida com payload 1KB (worst case WP admin password longo)', async () => {
    const plain = 'x'.repeat(1024);
    const cipher = await sealEncrypt(plain, kpA.publicKey);
    expect(await sealDecrypt(cipher, kpA.publicKey, kpA.privateKey)).toBe(plain);
  });

  it('cipher output é base64 não-vazio com tamanho ≥ plaintext + 48 bytes overhead', async () => {
    const cipher = await sealEncrypt('hi', kpA.publicKey);
    expect(cipher).toMatch(/^[A-Za-z0-9+/=]+$/);
    expect(cipher.length).toBeGreaterThan(40);  // 2 chars + 48 bytes sealed box overhead → base64 ~67 chars
  });
});
```

## Edge Cases

1. **Empty plaintext (`sealEncrypt('', pubKey)`):** libsodium aceita (cipher = só overhead 48 bytes). Roundtrip volta `''`. Documentar como behavior aceito (não bloquear).
2. **Plaintext com null bytes (`'foo\x00bar'`):** libsodium binary-safe, mas conversão `to_string` pode truncar em null. Test edge case → garantir que tokens Hostinger não contêm null (são strings ASCII printable).
3. **Cipher tampered (1 byte modificado):** sealed box detecta via MAC e rejeita. Validar via test (modificar último char base64 + tentar decrypt).
4. **Public key tampered:** rejeita também (não é a key correta pra esse cipher).
5. **`sodium.ready` não aguardado:** chamada antes do init lança error `Sodium not loaded`. Mitigação: `init()` helper interno que aguarda `sodium.ready` once.
6. **Keypair rotation futuro (Onda 1.5):** ADR-0008a §3.3 descreve. NÃO implementar agora; só deixar comment no doc-comment.
7. **Env var malformada (base64 inválido):** sodium.from_base64 lança error. Boot validation (AC-3) deve detectar.
