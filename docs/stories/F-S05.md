## Story

Como **dev backend (Dex)**, eu quero **8 endpoints Hono REST implementados em `tracking-api/src/routes/`**, pra que **o frontend Vite SPA consiga criar/listar/atualizar hosting accounts + sites + installations via HTTP padrão** (Bearer JWT do Supabase auth).

## Acceptance Criteria

### AC-1 — POST /api/hosting-accounts cria conta + valida token

- **Given** que Diego cola token Hostinger em `/sites/connect`
- **When** UI POST com body `{ provider: 'hostinger', token: '...', label: 'Diego pessoal', wp_admin_password?: '...' }`
- **Then**:
  1. `sealEncrypt(token, PUB_KEY)` (F-S02)
  2. `getProvider('hostinger', { token }).pingToken()` (F-S04) — se 401, retorna `401 { error: { code: 'INVALID_TOKEN', message: 'Token Hostinger inválido ou revogado' }}`
  3. Se OK, `storage.createAccount({ ...input, token_encrypted: cipher })` (F-S01)
  4. Retorna `201 { data: { id, status: 'active', created_at } }` — NÃO inclui `token_encrypted` no response.

### AC-2 — GET /api/hosting-accounts lista do tenant

- **Given** que Diego pode ter múltiplas accounts (MVP típico = 1)
- **When** GET sem query params
- **Then**:
  1. `storage.listAccounts({ tenant_id: MENTORIA_TENANT_ID })` (hardcoded const)
  2. Map cada account removendo `token_encrypted`
  3. Retorna `200 { data: HostingAccount[] }`

### AC-3 — DELETE /api/hosting-accounts/:id remove conta

- **Given** que token vazado precisa ser removido sem perder audit
- **When** DELETE
- **Then**:
  1. `storage.deleteAccount(id)` remove `gtm:account:<id>` HASH + SREM `gtm:account:list`
  2. **NÃO toca** `gtm:audit:*` keys (preserva log histórico per ADR-0008a §2.3)
  3. Retorna `204 No Content`

### AC-4 — GET /api/sites lista merged provider + storage

- **Given** que UI precisa ver sites Hostinger + status installation
- **When** GET (após Diego conectar account)
- **Then**:
  1. Pra cada account ativo: `provider.listSites()` (cache 60s em memória)
  2. Pra cada site: `storage.getInstallationBySite(site.domain)` (idempotency lookup)
  3. Merge → array `Site[]` com campos: `{ domain, wp_version, php_version, ttfb_ms, is_wordpress, status?: 'installed' | 'draft' | 'failed' | 'drift_detected', brand_slug?, container_id?, last_install_at?, installation_id? }`
  4. Retorna `200 { data: EnrichedSite[] }`

### AC-5 — POST /api/installations cria draft

- **Given** que Diego escolheu brand pra site específico
- **When** POST `{ hosting_account_id, site_domain, brand_slug }`
- **Then**:
  1. Validar `brand_slug ∈ {'mentoria','mentoria-app','zerohum','ifrn'}` (Zod)
  2. Backend lookup `gtm_container_id` correto **por brand_slug hardcoded MVP** (CLAUDE.md "Brands rastreadas"):
     - mentoria → `GTM-5J587HS3`
     - mentoria-app → `GTM-KMK749ZW`
     - zerohum → `GTM-WVWQVMP`
     - ifrn → `GTM-5J587HS3` (reusa Mentoria)
  3. `storage.createInstallation({ tenant_id, hosting_account_id, site_domain, brand_slug, gtm_container_id, plugin_version: 'gtm4wp-1.18+bootstrap-v1', status: 'draft', attempt_count: 0, created_by: ctx.user.id })`
  4. **Idempotência:** se já existir installation pro mesmo domain → retorna existing (não cria novo) — F-S01 AC-4 cobre.
  5. Retorna `201 { data: GtmInstallation }`

### AC-6 — POST /api/installations/:id/deploy adquire lock + dispara async

- **Given** que deploy é operação longa (10-30s)
- **When** POST `/api/installations/:id/deploy`
- **Then**:
  1. `storage.acquireLock(id, 60)` — se false: `409 { error: { code: 'DEPLOY_IN_PROGRESS', message: 'Outro deploy em andamento pra esse site' }}`
  2. Se OK: dispara worker async via `setImmediate(() => deployJob(id))` OR worker thread
  3. Worker:
     - `storage.updateInstallation(id, { status: 'uploading', attempt_count++, last_attempted_at })`
     - Cria pluginPath via F-S13 `buildPlugin(installation)`
     - `provider.deployPlugin({ domain, slug, pluginPath })` (F-S04)
     - Cleanup pluginPath
     - Pub event SSE `gtm:events:<id>` LIST (F-S12)
     - Após upload: `storage.updateInstallation(id, { status: 'uploaded_pending_activation', upload_dir_name })`
     - Tentar activation (best-effort fallback D per ADR-0008 §3.4 — UI explicita se falha)
     - `validate(domain, expectedContainerId)` (F-S06)
     - `storage.updateInstallation(id, { status: 'installed' | 'failed', last_validation_result })`
     - `releaseLock(id)`
  4. Retorna **202 Accepted** `{ data: { job_id: id, sse_url: '/api/installations/:id/events' } }`

### AC-7 — GET /api/installations/:id retorna status atual

- **Given** que UI faz polling fallback se SSE falhar
- **When** GET
- **Then** retorna `200 { data: GtmInstallation }` com status atual + `last_validation_result` se presente. Idempotente. Cache no-store (Cache-Control header).

### AC-8 — POST /api/installations/:id/revalidate re-roda validador

- **Given** que Diego clica "Revalidar" pra detectar drift
- **When** POST
- **Then**:
  1. NÃO precisa lock (revalidação é read-only no site, write apenas no Redis)
  2. `validate(domain, expectedContainerId)` (F-S06)
  3. `storage.updateInstallation(id, { last_validation_at, last_validation_result, status: result.passed ? 'installed' : 'drift_detected' })`
  4. Append audit `validation_passed` ou `validation_failed`
  5. Retorna `200 { data: { passed: boolean, stage, details } }`

### AC-9 — DELETE /api/installations/:id marca uninstalled

- **Given** que actual WP cleanup é Onda 1.5 (per ADR-0008 §6 Story 8)
- **When** DELETE
- **Then**:
  1. `storage.updateInstallation(id, { status: 'uninstalled' })` — soft delete
  2. NÃO chama provider.deletePlugin (Onda 1.5)
  3. Append audit `uninstalled`
  4. Retorna `200 { data: { status: 'uninstalled' } }` + warning header `X-Onda: "Cleanup WP filesystem é Onda 1.5"`

### AC-10 — JSON response shape consistente

- **Given** que UI precisa parsing previsível
- **When** qualquer endpoint retorna
- **Then** shape:
  - Sucesso: `{ data: T }`
  - Erro: `{ error: { code: 'STRING_UPPER_SNAKE', message: 'string PT-BR', request_id: 'uuid' }}`
  - `request_id` gerado por middleware (UUID v4 por request) — propagado em response header `X-Request-ID` também pra correlação logs.

### AC-11 — Auth middleware JWT Bearer

- **Given** que Supabase auth já existe no tracking-app
- **When** request sem `Authorization: Bearer <jwt>` header OU JWT inválido
- **Then** middleware retorna `401 { error: { code: 'UNAUTHORIZED', message: 'Autenticação necessária' }}`. Middleware valida JWT via `supabase.auth.getUser(jwt)` ou jwt.verify com `SUPABASE_JWT_SECRET` env.

## Tech Notes

**Decisões herdadas:**
- ADR-0008 §6 Story 3 lista endpoints — F-S05 espelha.
- UX §10.5 canonical contract — endpoints alinham.
- **River Q1 decisão: REST clássico** — `POST /hosting-accounts`, `GET /sites`, sub-actions `POST /installations/:id/deploy`. Híbrido onde action ≠ CRUD.

**Patterns a usar:**
- Hono routing: 1 file por resource (`routes/hosting-accounts.ts`, `routes/sites.ts`, `routes/installations.ts`).
- Zod schemas pra body validation (`zod` já no `tracking-api` provavelmente).
- Error middleware central em `src/middleware/error.ts` — captura throws + mapeia pra JSON `{ error: ... }`.
- Auth middleware em `src/middleware/auth.ts` — apenas em rotas `/api/*` protegidas.
- `setImmediate` simples pra MVP (1 réplica) — Onda 1.5 considerar BullMQ/Queue.
- Const `MENTORIA_TENANT_ID` em `src/lib/constants.ts` (hardcoded UUID per CLAUDE.md core.schools slug='mentoria').
- Const `BRAND_GTM_MAP` lookup em mesma file.

**LoC estimado:** ~600 (8 routes + Zod + auth + error middleware + worker dispatcher).

**Performance:**
- Endpoints sync (não-deploy): <100ms p95.
- POST /deploy: <200ms (só enfileira, deploy é async).
- Worker deploy: 15-60s típico.

## Edge Cases

1. **Concurrent POST /api/installations 2× mesmo domain:** F-S01 idempotency cobre — retorna mesmo ID.
2. **Token expirado pós-create:** `pingToken()` antes de deploy detecta, marca `account.status='token_expired'`, retorna 401 com mensagem clara.
3. **Worker crash mid-deploy:** lock TTL 60s libera. Storage installation fica em `uploading` — Diego pode reset via revalidate OU UI mostra warning "deploy travado".
4. **JWT expirado:** middleware retorna 401, frontend redireciona login.
5. **brand_slug inválido:** Zod rejeita 422.
6. **Auth user diferente do install owner:** MVP single-tenant, todos vêem tudo. RLS NÃO se aplica (sem DB). Onda 2 implementa.
7. **DELETE durante deploy ativo:** marca uninstalled mesmo se uploading — audit registra discrepancy.
