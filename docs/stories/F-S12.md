## Story

Como **dev fullstack (Dex)**, eu quero **endpoint SSE `GET /api/installations/:id/events` no backend + cliente EventSource no frontend hook `useInstallTracking`**, pra que **o modal de progresso (F-S09 InstallProgressModal) atualize passo-a-passo em tempo real** (não polling).

## Acceptance Criteria

### AC-1 — Endpoint SSE Hono retorna text/event-stream

- **Given** que F-S05 deploy é async + UI precisa progress real-time
- **When** Dex modifica `tracking-api/src/routes/installations.ts`
- **Then** rota `GET /api/installations/:id/events`:
  - Validates auth JWT (middleware F-S05)
  - Retorna `Content-Type: text/event-stream` + `Cache-Control: no-cache` + `Connection: keep-alive`
  - Body é `ReadableStream` formato SSE:
    ```
    data: {"step":"upload_started","status":"in_progress"}\n\n
    data: {"step":"upload_complete","status":"done","timing_ms":12300}\n\n
    ```
  - Uses `hono/streaming` helper `c.streamSSE(handler)` ou implementação manual com `c.body(stream)` + `Content-Type` set explícito.

### AC-2 — Worker publica eventos durante deploy

- **Given** que worker F-S05 AC-6 dispara deploy async
- **When** worker transitiona step
- **Then** cada transition publica em Redis LIST `gtm:events:<installation_id>`:
  - `upload_started`, `upload_complete | upload_failed`
  - `activation_started`, `activation_complete | activation_failed`
  - `validation_passed | validation_failed`
  - JSON shape: `{ step: string, status: 'in_progress'|'done'|'failed', timing_ms?: number, error?: string }`
  - LIST TTL 5min via `EXPIRE gtm:events:<id> 300` (auto-cleanup)

### AC-3 — SSE endpoint consume Redis LIST via BRPOP

- **Given** que SSE precisa stream eventos do worker
- **When** client conecta `GET /events`
- **Then** endpoint loop:
  1. `BRPOP gtm:events:<installation_id> 15` (timeout 15s) — bloqueia até evento OR timeout
  2. Se evento: emit SSE `data: <json>\n\n` pro client
  3. Se timeout: emit heartbeat `: ping\n\n` (comment SSE) + continue loop
  4. Se evento step === `installed` ou `failed`: close stream
  5. Se client disconnect (controller.abort): close stream + return

Implementação alternativa: Redis Streams `XREAD BLOCK 15000 STREAMS gtm:events:<id> $` (mais robusto, suporta multi-consumer). Dex decide simplicidade vs robustness — F-S12 AC-3 nota inicial é LIST/BRPOP, Streams aceito se Dex preferir.

### AC-4 — Heartbeat ping cada 15s previne proxy timeout

- **Given** que Caddy/Traefik Easypanel pode timeout conexão idle
- **When** sem eventos por 15s
- **Then** server emit `: ping\n\n` (comment line SSE, ignored by EventSource) → mantém connection viva. Cliente browser ignora silenciosamente.

### AC-5 — Cliente EventSource em useInstallTracking

- **Given** que F-S11 useInstallTracking.start() inicia subscription
- **When** `start(brandSlug)` chamado
- **Then**:
  ```ts
  const eventSource = new EventSource(`/api/installations/${installationId}/events`, {
    withCredentials: true,
  });
  eventSource.onmessage = (e) => {
    const data = JSON.parse(e.data);
    setProgress({ step: data.step, status: data.status, timing_ms: data.timing_ms });
    if (data.step === 'installed') setStatus('installed');
    if (data.step === 'failed') setStatus('failed');
  };
  eventSource.onerror = () => {
    // Fallback polling (AC-6)
  };
  // cleanup
  return () => eventSource.close();
  ```

### AC-6 — Fallback polling em error

- **Given** que EventSource pode falhar (proxy block, navegador antigo)
- **When** `onerror` dispara
- **Then** hook degrada pra `setInterval(() => fetch('/api/installations/:id'), 2000)` — atualiza status a cada 2s. Loop para quando status === installed | failed.

### AC-7 — Vitest tests cobrem ambos paths

- **Given** que SSE é stateful
- **When** roda
- **Then**:
  - Server: mock Redis BRPOP retornando eventos → assert SSE format correto na response stream
  - Client: mock EventSource (lib `eventsource-mock`) → simulate events → assert progress state updates
  - Fallback polling testado com EventSource desabilitado (mock onerror)

**Patterns a usar:**
- `hono/streaming` import: `import { streamSSE } from 'hono/streaming'`
- SSE format: `data: <json>\n\n` (2 \n exigidos)
- Heartbeat format: `: <comment>\n\n` (colon prefix = comment)
- LIST escolha simples MVP; Streams = Onda 1.5 se Dex quer multi-consumer support

- EventSource native browser (nenhum lib npm necessário no frontend)

## Edge Cases

1. **Worker crash mid-deploy:** Redis LIST tem eventos parciais. SSE stream stops, client EventSource detect (sem mais eventos por 60s) → onerror → fallback polling → final status reflete `uploading` (no `installed`). UI mostra warning.
2. **2 clients SSE no mesmo installation_id:** com BRPOP, primeiro client pega cada evento (LIST consume). Segundo client não vê. Mitigação: usar Redis Pub/Sub (PUBLISH/SUBSCRIBE) que broadcast — mais robusto multi-client. Dex decide MVP single-client vs pub/sub.
3. **Heartbeat em Caddy buffer:** se proxy bufeira 8KB, 15s ping (~10 bytes) pode acumular. Felix valida buffering off.
4. **EventSource não suportado:** IE11. MVP Diego Chrome/Firefox/Safari moderno — aceito.
5. **TTL Redis events 5min expira durante install lento:** ajustar pra 10min se workers >5min observados. ADR-flag mid-stream.
6. **Authorization header EventSource:** browser NÃO suporta custom headers em EventSource. Workarounds: cookie auth OR query param token (`?token=<jwt>`). Aceito MVP cookie session OU query param com warning logs.
