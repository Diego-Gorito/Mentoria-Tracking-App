## Story

Como **dev frontend (Dex)**, eu quero **4 custom hooks encapsulando data fetching + state management** dos endpoints F-S05, pra que **pages F-S10 não duplicarem lógica de loading/error/cache** (DRY + testabilidade).

## Acceptance Criteria

### AC-1 — useSites returns sites + isLoading + error + refresh

- **Given** que SitesListPage precisa fetch + cache + refresh manual
- **When** Dex cria `tracking-app/src/hooks/useSites.ts`
- **Then**:
  ```ts
  export function useSites(): {
    sites: EnrichedSite[];
    isLoading: boolean;
    error: Error | null;
    refresh: () => Promise<void>;
  }
  ```
  - Fetch `GET /api/sites` no mount
  - Cache 60s in-memory (React Query OR custom Map<Cache, ttl>)
  - `refresh()` força refetch (invalida cache)
  - Error messages traduzidos PT-BR (UX §4.6 map):
    - Network error → "Sem conexão com o servidor"
    - 401 → "Sessão expirada — faça login novamente"
    - 500 → "Erro no servidor. Tente novamente em instantes"

### AC-2 — useHostingerAccount returns account + actions

- **Given** que ConnectHostingerPage precisa connect + disconnect
- **When** Dex cria `tracking-app/src/hooks/useHostingerAccount.ts`
- **Then**:
  ```ts
  export function useHostingerAccount(): {
    account: HostingAccount | null;
    isConnected: boolean;
    connect: (token: string, label?: string, wpAdminPass?: string) => Promise<void>;
    disconnect: () => Promise<void>;
    isConnecting: boolean;
    connectError: Error | null;
  }
  ```
  - `connect()` → `POST /api/hosting-accounts` body `{ provider: 'hostinger', token, label, wp_admin_password? }`
  - `disconnect()` → `DELETE /api/hosting-accounts/:id` (id do account atual)
  - `account` carrega via `GET /api/hosting-accounts` no mount (1ª item)
  - `isConnecting` flag durante POST
  - `connectError` em case 401 (token inválido) ou outros

### AC-3 — useInstallTracking(siteId) com SSE subscribe

- **Given** que F-S12 implementa SSE
- **When** Dex cria `tracking-app/src/hooks/useInstallTracking.ts`
- **Then**:
  ```ts
  export function useInstallTracking(siteId: string): {
    install: GtmInstallation | null;
    progress: { step: string; status: 'in_progress' | 'done' | 'failed'; timing_ms?: number };
    status: 'idle' | 'installing' | 'installed' | 'failed';
    result: ValidationResult | null;
    start: (brandSlug: BrandSlug) => Promise<void>;
  }
  ```
  - `start(brandSlug)` → `POST /api/installations` + `POST /api/installations/:id/deploy` + abre `EventSource(/api/installations/:id/events)` (F-S12)
  - SSE event handler atualiza `progress` step-by-step
  - Final event `installed` ou `failed` atualiza `status`
  - Cleanup unsubscribe SSE em useEffect cleanup
  - Fallback polling F-S12 AC-6 se EventSource error

### AC-4 — useAuditLog(siteId?) returns entries

- **Given** que SiteDetailPage + SiteAuditLogPage consomem
- **When** Dex cria `tracking-app/src/hooks/useAuditLog.ts`
- **Then**:
  ```ts
  export function useAuditLog(siteId?: string): {
    entries: InstallationAudit[];
    isLoading: boolean;
    error: Error | null;
    refresh: () => Promise<void>;
  }
  ```
  - Se `siteId` → `GET /api/sites/:id/audit-log` (assumindo F-S05 expõe ou alias `/installations/:installation_id/audit`)
  - Se undefined → global (MVP carrega últimas 50 cross-installation)
  - Pagination cursor-based em Onda 1.5 (MVP só primeiras 50)

### AC-5 — Stack data fetching React Query OR custom

- **Given** que `tracking-app` pode já ter @tanstack/react-query
- **When** Dex inicia
- **Then**:
  - Se já presente → reuse (gerencia cache, retry, stale-while-revalidate)
  - Se ausente → custom hook com `useState` + `useEffect` + ref pra cancel + Map cache TTL
  - Dex decide e documenta no PR description

### AC-6 — Error handling PT-BR

- **Given** que UX §4.6 lista mensagens
- **When** fetch falha
- **Then** hooks retornam `error: Error` com `.message` traduzida PT-BR via helper `translateApiError(err): string`:
  - `Network failed` → "Sem conexão com o servidor"
  - `401` → "Sessão expirada"
  - `403` → "Sem permissão"
  - `404` → "Recurso não encontrado"
  - `409` → "Outro deploy em andamento" (lock conflict)
  - `500+` → "Erro no servidor — tente novamente"

### AC-7 — Vitest tests

- **Given** que mock fetch é fácil em vitest
- **When** roda
- **Then** 4 hooks × 3 cenários (idle→loading→success, idle→loading→error, refresh re-fetch) = 12 testes.

## Test Plan

### Unit (vitest + RTL `renderHook`)

```ts
import { renderHook, waitFor } from '@testing-library/react';
import { useSites } from '../useSites';

vi.mock('global', () => ({ fetch: vi.fn() }));

it('useSites: idle → loading → success', async () => {
  vi.mocked(fetch).mockResolvedValueOnce({
    ok: true, json: () => Promise.resolve({ data: [{ domain: 'x.com' }] })
  } as any);

  const { result } = renderHook(() => useSites());
  expect(result.current.isLoading).toBe(true);

  await waitFor(() => expect(result.current.isLoading).toBe(false));
  expect(result.current.sites).toHaveLength(1);
});

it('useSites: error retornado em PT-BR', async () => {
  vi.mocked(fetch).mockRejectedValueOnce(new Error('Network failed'));
  const { result } = renderHook(() => useSites());
  await waitFor(() => expect(result.current.error).not.toBeNull());
  expect(result.current.error?.message).toMatch(/Sem conexão/);
});
```

## Edge Cases

1. **Component unmount mid-fetch:** AbortController cancela request, evita warning "setState on unmounted".
2. **Token expirou:** 401 → translateApiError → "Sessão expirada" → AuthGuard pega no próximo render.
3. **Concurrent connect calls:** isConnecting flag bloqueia 2× submit.
4. **SSE reconnect (network blip):** EventSource auto-reconecta (browser native). Hook re-subscribes em onerror.
5. **useInstallTracking start() chamado 2×:** segundo call retorna 409 → translateApiError → "Outro deploy em andamento" toast.
