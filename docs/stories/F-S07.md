## Story

Como **dev backend (Dex)**, eu quero **helper `safeAuditPayload(raw)` sanitizar payloads antes de gravar em `gtm:audit:*`**, pra que **logs nunca incluam tokens / passwords / response bodies brutos** — LGPD-safe by default.

## Acceptance Criteria

### AC-1 — Função safeAuditPayload exportada

- **Given** que callers (F-S04 adapter, F-S05 endpoints) vão construir payloads
- **When** Dex cria `tracking-api/src/lib/audit.ts`
- **Then** exporta:
  ```ts
  export interface SafePayload {
    site_domain?: string;
    status_code?: number;
    timing_ms?: number;
    file_count?: number;
    upload_dir_name?: string;
    error_summary?: string;       // ≤500 chars
    retry_attempt?: number;
    [key: string]: unknown;        // permite extension, filtrado por whitelist
  }

  export function safeAuditPayload(raw: Record<string, unknown>): SafePayload;
  ```

### AC-2 — Whitelist explícita aplica

- **Given** que ADR-0008 §3.7 lista keys aceitas
- **When** `safeAuditPayload({ site_domain: 'x.com', status_code: 200, timing_ms: 350, file_count: 12, upload_dir_name: 'gtm4wp-abc', error_summary: 'short err', retry_attempt: 2 })` chamada
- **Then** retorna objeto com EXATAMENTE essas 7 keys preservadas (todas pertencem à whitelist).

### AC-3 — Blacklist rejeita keys sensíveis silenciosamente

- **Given** que callers podem por engano passar `token`, `password`, etc.
- **When** `safeAuditPayload({ token: 'abc123', password: 'foo', secret: 'bar', bearer: 'xyz', authorization: 'Auth: Bearer', api_key: 'k', site_domain: 'x.com' })` chamada
- **Then** retorna **apenas** `{ site_domain: 'x.com' }` — todas 6 keys sensíveis removidas silenciosamente (sem throw, sem warning visível). Comportamento defensivo by-default.

### AC-4 — error_summary truncado 500 chars

- **Given** que stack traces podem ser longos
- **When** `safeAuditPayload({ error_summary: 'x'.repeat(1000) })` chamada
- **Then** retorna `error_summary` exatamente 500 chars (`'x'.repeat(500)`). Sem reticências ou marcador.

### AC-5 — Wrapper appendAuditWithSanitization centraliza

- **Given** que endpoints/adapter NÃO devem chamar `storage.appendAudit` direto
- **When** Dex cria wrapper:
  ```ts
  export async function appendAuditWithSanitization(
    storage: IGtmStorage,
    input: {
      installation_id: InstallationId;
      tenant_id: TenantId;
      action: InstallationAudit['action'];
      rawPayload?: Record<string, unknown>;
      actor_user_id?: string;
      actor_source: InstallationAudit['actor_source'];
    }
  ): Promise<void>
  ```
- **Then** internamente faz `storage.appendAudit({ ...input, payload: safeAuditPayload(input.rawPayload ?? {}) })`. Callers em F-S04 + F-S05 usam ESSE wrapper, NÃO `storage.appendAudit` direto.

### AC-6 — Blacklist recursivo em nested objects

- **Given** que dev pode passar payload nested `{ raw: { hostinger_token: 'abc' }}`
- **When** safeAuditPayload com objeto profundo
- **Then** blacklist aplica em CADA nível — se key `hostinger_token` matches blacklist pattern (`token`), remove. Função walks tree depth ≤3 (proteção contra infinite recursion via circular refs).

### AC-7 — Vitest tests cobrem cenários

- **Given** que function pure é fácil testar
- **When** roda vitest
- **Then** 5 cenários green:
  1. Whitelist pass-through: 7 keys aceitas preservadas
  2. Blacklist reject: 6 keys sensíveis removidas, 1 inocente preservada
  3. error_summary truncate: 1000 chars → 500 chars
  4. Nested blacklist: `{ raw: { hostinger_token: 'x' }}` → `{ raw: {} }` (token removido nested)
  5. Mixed: `{ site_domain: 'x', token: 'leak', timing_ms: 100 }` → `{ site_domain: 'x', timing_ms: 100 }`

## Tech Notes

**Decisões herdadas:**
- ADR-0008 §3.7 define whitelist + blacklist exatos — copiar literal.
- ADR-0008a §2.4 define `InstallationAudit.payload: Record<string, unknown>` (sem schema rígido — wrapper guarantee).
- Pattern existing em `tracking-api/src/lib/safeLog.ts` (se existe — Dex verifica) — pode reuso.

**Patterns a usar:**
- Whitelist explícita (set/array literal):
  ```ts
  const ALLOWED_KEYS = new Set([
    'site_domain', 'status_code', 'timing_ms', 'file_count',
    'upload_dir_name', 'error_summary', 'retry_attempt',
  ]);
  ```
- Blacklist regex (case-insensitive):
  ```ts
  const SENSITIVE_PATTERN = /token|password|secret|bearer|authorization|api[_-]?key/i;
  ```
- Recursion limited depth=3, plain object check (`typeof === 'object' && !Array.isArray && constructor === Object`).
- Pure function — sem side effects, sem async (sanitization é sync; storage.appendAudit é async).

**LoC estimado:** ~80 (audit.ts 50 + tests 60).

**Onda 1.5 migration:** quando Supabase storage entrar, audit vira `INSERT INTO tracking.installation_audit (...)` — wrapper `appendAuditWithSanitization` permanece same (interface IGtmStorage troca impl). pgTAP test (ADR-0008 §3.7) substitui vitest test no Onda 1.5 — payload NÃO contém keys `token|password|secret|bearer` validado em DB row direto.

## Test Plan

### Unit (vitest)

```ts
import { describe, it, expect } from 'vitest';
import { safeAuditPayload, appendAuditWithSanitization } from '../audit';

describe('safeAuditPayload', () => {
  it('whitelist preserva 7 keys aceitas', () => {
    const r = safeAuditPayload({
      site_domain: 'x.com', status_code: 200, timing_ms: 350,
      file_count: 12, upload_dir_name: 'd', error_summary: 'e', retry_attempt: 1,
    });
    expect(Object.keys(r).sort()).toEqual([
      'error_summary', 'file_count', 'retry_attempt', 'site_domain',
      'status_code', 'timing_ms', 'upload_dir_name'
    ]);
  });

  it('blacklist remove 6 keys sensíveis', () => {
    const r = safeAuditPayload({
      token: 'x', password: 'y', secret: 'z',
      bearer: 'a', authorization: 'b', api_key: 'c',
      site_domain: 'safe.com',
    });
    expect(r).toEqual({ site_domain: 'safe.com' });
    expect(r).not.toHaveProperty('token');
  });

  it('error_summary trunca 1000 → 500', () => {
    const r = safeAuditPayload({ error_summary: 'x'.repeat(1000) });
    expect(r.error_summary).toHaveLength(500);
  });

  it('nested blacklist remove hostinger_token nested', () => {
    const r = safeAuditPayload({
      site_domain: 'x.com',
      meta: { hostinger_token: 'leak', other: 'safe' }
    } as any);
    expect((r as any).meta?.hostinger_token).toBeUndefined();
    expect((r as any).meta?.other).toBe('safe');
  });

  it('mixed: keep safe + drop sensitive', () => {
    const r = safeAuditPayload({
      site_domain: 'x.com', token: 'leak', timing_ms: 100,
    });
    expect(r).toEqual({ site_domain: 'x.com', timing_ms: 100 });
  });
});

describe('appendAuditWithSanitization', () => {
  it('chama storage.appendAudit com payload sanitizado', async () => {
    const storage = { appendAudit: vi.fn() };
    await appendAuditWithSanitization(storage as any, {
      installation_id: 'i' as any, tenant_id: 't' as any,
      action: 'upload_complete', actor_source: 'tracking-api',
      rawPayload: { token: 'leak', site_domain: 'x.com' },
    });
    expect(storage.appendAudit).toHaveBeenCalledWith(expect.objectContaining({
      payload: { site_domain: 'x.com' },
    }));
  });
});
```

## Edge Cases

1. **Circular reference no rawPayload:** `JSON.stringify` throws. Mitigação: try/catch wrap; depth limit=3 protege.
2. **Number muito grande (`Number.MAX_SAFE_INTEGER`):** OK, JSON suporta. NÃO sanitiza.
3. **Symbol key:** ignorado (whitelist é string-keyed).
4. **Class instance passed:** `constructor === Object` check rejeita, retorna `{}`.
5. **Empty rawPayload `{}`:** retorna `{}`. OK.
6. **`undefined` value em allowed key:** preserva (não filtra undefined).
7. **Key case-sensitive:** `Token` (capital T) NÃO matches blacklist regex `/token/i`? **SIM matches** (flag `i`). Cobertura segura.
