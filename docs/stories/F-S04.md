## Story

Como **dev backend (Dex)**, eu quero **`HostingerAdapter` totalmente implementado** invocando o Hostinger MCP server (`mcp__hostinger__*` primitives), pra que **endpoints `/api/sites` + `/api/installations/:id/deploy` deleguem operações reais ao provider sem conhecer detalhes do MCP**.

## Acceptance Criteria

### AC-1 — `listSites()` normaliza resposta MCP

- **Given** que MCP `mcp__hostinger__hosting_listWebsitesV1` retorna array de websites
- **When** `adapter.listSites()` é chamado
- **Then** retorna `Site[]` mapeado per F-S03 AC-2:
  - `domain` ← `website.domain`
  - `wp_version` ← detectado via response (se MCP expõe) ou `undefined`
  - `php_version` ← `website.php_version` se disponível
  - `is_wordpress` ← `true` se response indica WP (campo `is_wordpress` ou heuristic)
- Limite: páginate até final OU primeiras 100 sites (whichever first) — MVP Diego tem <20 sites.

### AC-2 — `verifyDomain(domain)` consulta lista

- **Given** que validação prévia evita deploy errado
- **When** `verifyDomain('zerohum.com.br')` chamado
- **Then** retorna `true` se domain aparece em `listSites()`, `false` caso contrário. Implementação pode cachear `listSites()` por 60s em memória (Map TTL) pra evitar N chamadas redundantes em mesmo request flow.

### AC-3 — `deployPlugin(opts)` invoca MCP com payload correto

- **Given** que MCP espera `{ domain, slug, pluginPath }`
- **When** `deployPlugin({ domain: 'zerohum.com.br', slug: 'gtm4wp-mentoria', pluginPath: '/tmp/build-abc/' })` chamado
- **Then** Adapter chama `mcp__hostinger__hosting_deployWordpressPlugin` (via MCP client lib OU REST direto se Dex preferir simplicidade) com exatamente esse payload. Retorna `DeployResult` mapeado:
  - `status` ← `'success'` se `mcp.summary.failed === 0`, `'partial'` se `failed > 0 && successful > 0`, `'failed'` se todos falham
  - `summary` ← `{ successful, failed }` literal
  - `uploadDirName` ← MCP retorna (com random suffix)
  - `errorSummary` ← extraído de `mcp.error` truncado 500 chars

### AC-4 — `pingToken()` detecta token expirado

- **Given** que ADR-0008 §3.8 diz: chamar `listSites()` com page=1 per_page=1 como ping
- **When** `pingToken()` chamado
- **Then**:
  - Retorna `true` se MCP responde 200 (token válido)
  - Retorna `false` se MCP retorna 401 (token revogado/expirado)
  - **Propaga** error em 5xx ou network (não silencia — caller decide retry)

### AC-5 — Retry exponencial em 5xx + ECONNRESET

- **Given** que ADR-0008 §3.9 define backoff `[1000ms, 2000ms, 4000ms]`
- **When** `deployPlugin` falha com `status=503`
- **Then** wrapper `withRetry<T>(fn)`:
  1. Tenta 1× — falha
  2. `sleep(1000)` + tenta 2× — se sucesso, retorna
  3. `sleep(2000)` + tenta 3× — se sucesso, retorna
  4. `sleep(4000)` + tenta 4× — se ainda falha, throw original error
  5. 4xx (401, 403, 404, 422) = **fail-fast** sem retry
- Total worst case: ~7s antes de propagar error.

### AC-6 — Audit log em cada retry attempt

- **Given** que F-S01 storage tem `appendAudit`
- **When** retry acontece (attempt 2, 3, 4)
- **Then** Adapter chama `storage.appendAudit({ action: 'upload_started', actor_source: 'tracking-api', payload: { retry_attempt: N, status_code: prevError.status, timing_ms: ... } })` antes de cada retry. Final result (success ou failure) também loga audit `upload_complete` ou `upload_failed`.

### AC-7 — Unit tests cobrem cenários happy + edge

- **Given** que MCP é externo, precisa ser mockado
- **When** Dex roda vitest
- **Then** 6 cenários green:
  1. `listSites()` happy → retorna 3 Sites normalizados
  2. `pingToken()` 200 → true
  3. `pingToken()` 401 → false (sem throw)
  4. `deployPlugin` happy → `DeployResult { status: 'success', summary: {...} }`
  5. `deployPlugin` 503 com retry → 3 tentativas mockadas, 3ª retorna 200, final `success`
  6. `deployPlugin` 4xx fail-fast → 1 tentativa só, throw imediato

## Tech Notes

**Decisões prévias herdadas:**
- ADR-0008 §3.1 descobriu via investigação MCP source: `hosting_deployWordpressPlugin` aceita `pluginPath` (directory) + adiciona random suffix em `uploadDirName`. **Idempotência mora em F-S01 storage** (`gtm:install:by_site:*`), NÃO no filesystem.
- ADR-0008 §3.9 define retry só em 5xx + network, NÃO em 4xx (auth/validation).
- Constructor armazena `token` (já decifrado pelo caller — F-S05 chama `sealDecrypt` antes).

**Patterns a usar:**
- `withRetry<T>(fn: () => Promise<T>, attempts = 3, backoff = [1000, 2000, 4000])` como utility separado em `tracking-api/src/lib/retry.ts` (reusável).
- Logger structured `console.log(JSON.stringify({ event, ... }))` — Easypanel captura stdout 7d retention (ADR-0008 §3.7).
- Em caso MCP server lib indisponível em Node runtime: usar REST direto `fetch('https://api.hostinger.com/api/hosting/v1/websites', { headers: { Authorization: 'Bearer ' + token }})` — endpoint público confirmado.
- **Caveat random suffix:** Dex deve documentar em comment que mesmo slug `gtm4wp-mentoria` vira `gtm4wp-mentoria-aB3kZ9pQ` na 1ª e `gtm4wp-mentoria-Xm7sJ2nL` na 2ª — orphans cleanup é Onda 1.5 (F-S15 runbook menciona).

**LoC estimado:** ~250 (impl 130 + retry wrapper 30 + MCP mock factory tests 90).

**Performance esperada:**
- `listSites()`: ~200-500ms (HTTP single Hostinger API)
- `deployPlugin()` happy: 10-30s (TUS upload N files)
- `pingToken()`: ~200-500ms
- Worst case retry exhaustion: ~7s extra + final error

## Edge Cases

1. **MCP retorna empty list `data: []`:** retorna `[]` válido — não throw. UI mostra "Nenhum site encontrado".
2. **Site não-WP retornado:** `is_wordpress: false` no Site type. UI mostra card "não suportado" (UX §10.2 + UX-015).
3. **TUS upload credentials expiram mid-deploy:** MCP gerencia internamente per ADR-0008 §3.1. Timeout backend default 120s.
4. **Plugin path inexistente:** `deployPlugin` falha cedo (MCP). 4xx fail-fast.
5. **Multiple concurrent deployPlugin:** F-S01 lock (`gtm:lock:install:*`) cobre. Adapter assume caller já tem lock.
6. **Network timeout durante retry:** `withRetry` deve respeitar `AbortController` se passed; senão usar timeout 30s padrão.
7. **`uploadDirName` random suffix:** documentar visível no audit log (`payload.upload_dir_name`) pra forensics.
